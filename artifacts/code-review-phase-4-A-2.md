# Code Review — Phase 4.A.2 (sandbox-verifiable atlas implementation)

## Verdict

APPROVE.

All sixteen verification points pass. The phase lands the deterministic
atlas pipeline end-to-end: ten primitive functions, seven recipes, the
indexed-PNG encoder + manifest serializer/parser, the generator, the
loader (with the pinned `DEV-` refusal + em-dash), the atomic
`__RULESET_VERSION__` flip in the Vite plugin, the `idiv`/`imod`
relocation to `src/core/intmath.ts`, the `streams.atlas(recipeId)`
accessor with the addendum-B8-pinned per-call invariant, and the
diagnostic-page atlas-preview UI. Final gates: 821/821 tests passing
(net +179 over phase 4.A.1's 642), lint clean, build green at
62.95 KB raw / 24.03 KB gzipped (well under the 75 KB cap), and
`npm run gen-atlas` reproduces `assets/atlas.png` byte-identically
(SHA-256 `d1b4a8b73d3e2c1b7cd70c26fe15a08faae5d91351d9e2e9a542ce71727b8d1a`
matches `ATLAS_DIGEST`, the `placeholder` preset's `expectedHash`,
and the manifest's `atlasBinaryHash` field).

The atomic-flip discipline holds: `vite.config.ts` and `vitest.config.ts`
no longer inject `__RULESET_VERSION__`; the plugin owns it
(`scripts/vite-plugin-atlas-binary-hash.mjs:204-208`) and produces the
derived 64-hex value `54dfaa9a405cc0efdc280dc1e67c4a596112480e6500c625aad0278ee12a0d62`
which appears in the dist bundle. There is no transient `4.A.1-pre-atlas-`
sentinel anywhere on disk or in `dist/` (asserted by
`tests/build/no-transient-sentinel.test.ts:25-37`).

Three non-blocking documentation drifts and a handful of cosmetic
suggestions are listed below.

(`src/core/intmath.ts`) with a re-export shim from
`src/mapgen/math.ts` for backward compatibility. The `EMPTY_SHA256`

## Non-blocking suggestions

1. **ARCHITECTURE.md doc/code drift on the loader path.**
   `docs/ARCHITECTURE.md:607` says the runtime atlas-loader lives
   at `src/render/atlas-loader.ts`. The actual file is at
   `src/atlas/loader.ts`. The functional behavior (the pinned
   refusal message, the U+2014 em-dash, the missing-atlas refusal,
   the SHA-256 hash check) is preserved exactly, but the path is
   wrong in the doc. A one-line edit to `ARCHITECTURE.md` to
   change the path reference would close the drift. (Putting the
   loader under `src/atlas/` is arguably the more natural
   placement — the loader's lint scope and dependencies are
   atlas-pipeline-internal — so prefer adjusting the doc rather
   than relocating the code.)

2. **ARCHITECTURE.md doc/code drift on the primitive set.**
   `docs/ARCHITECTURE.md:436-446` lists primitives `paletteIndex,
   solidFill, pixel, outlineRect, fillRect, scanline, dither2x2,
   radialGradient` (8 names, plus `paletteGradient` and
   `valueNoise2D` referenced separately = 10 total). The actual
   implementation `src/atlas/primitives.ts` and the canonical
   `decision-memo-phase-4.md:65-75` table both define the set as
   `paletteIndex, paletteSwap, paletteGradient, bayerThreshold,
   valueNoise2D, rectMask, circleMask, lineMask, columnShift,
   scanlineResidue` (10 names, plus the public `hash2D` helper).
   The memo is the canonical reference per CLAUDE.md, so the
   implementation is correct; ARCHITECTURE.md is the artifact that
   needs updating. (`paletteSwap`, `bayerThreshold`, `rectMask`,
   `circleMask`, `lineMask`, `columnShift`, `scanlineResidue` are
   the actual primitive names; `solidFill`, `pixel`, `outlineRect`,
   `fillRect`, `scanline`, `dither2x2`, `radialGradient` were an
   earlier-draft naming.)

3. **ARCHITECTURE.md doc/code drift on the recipe-ID regex.**
   `docs/ARCHITECTURE.md:465` pins
   `^atlas-recipe\.(cyberpunk)\.(tile|monster|item|player|npc)\.[a-z][a-z0-9-]*$`
   (5 categories); `decision-memo-phase-4.md:330` pins
   `^atlas-recipe\.(cyberpunk)\.(tile|monster|item|npc|ui|boss)\.[a-z][a-z0-9_-]{0,31}$`
   (6 categories without `player`, length cap, `_` allowed); the
   implementation `src/registries/atlas-recipes.ts:54-55` is
   `^atlas-recipe\.(cyberpunk)\.(tile|monster|item|player|npc|ui|boss)\.[a-z][a-z0-9-]*$`
   (7 categories with `player`, no length cap, no `_`). All seven
   currently-shipped recipe IDs match the implementation regex; the
   memo's length cap of 31 chars and `_` permission are not
   exercised by any current ID. This is a contract/doc/code drift
   that is *latent* but will surface when Phase 6/7 adds a recipe
   whose ID needs `_` (or longer than 31 chars). Pick one canonical
   form, propagate it, and assert it via a test. (Adding `player`
   as a category is reasonable for the v1 release; the rest is
   bookkeeping.)

4. **`src/atlas/loader.ts:118-125` JSON.parse cast trick.** The
   loader calls `(JSON as unknown as { parse(s: string):
   unknown }).parse(manifestText)` to sidestep the
   `determinism/no-float-arithmetic` lint rule's
   `JSON.parse` ban. The cast works but is fragile (a future
   refactor could break the alias chain). Two cleaner options:
   (a) emit an `// eslint-disable-next-line determinism/no-float-arithmetic`
   comment with a one-line justification (the loader is the
   data-ingestion boundary; `JSON.parse` is appropriate here);
   (b) scope the lint rule to interior modules only, exempting the
   `src/atlas/loader.ts` boundary file. Option (a) is the smaller
   change.

5. **EMPTY_SHA256 / PLACEHOLDER_RULESET_VERSION duplication
   between plugin and build-info.** Both literals appear in
   `scripts/vite-plugin-atlas-binary-hash.mjs:36-39` AND
   `src/build-info.ts:8,11-12`. The Vite plugin (loaded by
   vite.config.ts at config-load time) cannot transit
   `src/build-info.ts`'s `@noble/hashes/sha256` import graph, so
   the duplication is forced. A test that asserts byte-equality
   between the two literal pairs would close the audit-readability
   gap and fail loudly if either drifted.

6. **`RecipeContext` type lives in `src/atlas/recipes/floor.ts`.**
   Six other recipe files and `src/atlas/generate.ts` and
   `src/registries/atlas-recipes.ts` all import `RecipeContext`
   from `./floor` — making the floor recipe accidentally a
   "shared types" module. A `src/atlas/recipes/types.ts` (or
   `src/atlas/recipe-context.ts`) holding the type definition
   would be the cleaner home; cosmetic.

7. **`fflate`-version self-test assertion (addendum-2 implication).**
   Addendum-2 in `decision-memo-phase-4.md:3088-3132` mentioned
   "the cross-runtime self-test gains an `fflate`-version
   assertion." The current `atlas-encoder-cross-runtime` self-test
   relies on the encoder's golden hash to *implicitly* catch an
   `fflate` version drift — any fflate change would cascade through
   `IDAT` bytes and break the hash. An explicit
   `expect(fflateModuleVersion).toBe("0.8.2")` would be a
   complementary trip-wire. Cosmetic.

8. **vitest `define`-key fallback path.** `src/build-info.ts:14-39`
   uses `typeof __X__ !== "undefined" ? __X__ : <fallback>` for
   each define key. Vitest's `vite-node` substitutes literally
   *only* if the key is present in the resolved `define` block.
   The atlas plugin runs `recompute()` eagerly inside `config()`
   (`scripts/vite-plugin-atlas-binary-hash.mjs:202`), so the
   resolved config has the real values for both `vite build` and
   `vite-node` — confirmed by the loader's
   `buildLoaderEnv()` test at `loader.test.ts:184-191`
   (`expect(env.rulesetVersion).toMatch(/^[0-9a-f]{64}$/)` passes).
   The architect's note about `configResolved()` being too late is
   correctly addressed by the eager `recompute()` in `config()`.
   Sound. Concern was unfounded; flagging for awareness only so
   future plugin maintainers know the timing constraint.

## Atomic-flip soundness

Confirmed sound. The flip is atomic in three independent senses:
- **Repository state:** `assets/atlas.png` + `assets/atlas.json` +
  the `vite.config.ts` / `vitest.config.ts` removal of static
  `__RULESET_VERSION__` are all in the same uncommitted change set.
- **Build-pipeline state:** the plugin emits exactly two states
  (`hash + missing + ruleset`): `(EMPTY_SHA256, true,
  PLACEHOLDER_RULESET_VERSION)` when `assets/atlas.png` is absent,
  or `(real-hash, false, derived-64-hex)` when present. No third
  transient state is reachable at any stage of the build.
- **Bundle state:** `tests/build/no-transient-sentinel.test.ts:25-37`
  scans the built `dist/` for the forbidden `4.A.1-pre-atlas-`
  substring; assertion would fire if any developer accidentally
  introduced an interim sentinel.

## Atlas-preview UI testing strategy

Adequate for 4.A.2; Phase 4.B's Playwright matrix is the load-bearing
verification.

The atlas-preview UI in `src/main.ts:282-460` is not unit-tested
inside vitest (the JSDOM `<canvas>` API would not exercise the real
PNG decode path). The strategy delegates browser verification to
Playwright on the live deploy:
- `tests/e2e/diagnostic.spec.ts:164-175` asserts `#atlas-preview`
  is visible and `window.__ATLAS_PREVIEW__ === "ready"`.
- `:177-195` asserts `__ATLAS_PREVIEW_BUILD_HASH__ ===
  __ATLAS_PREVIEW_LIVE_HASH__ === ATLAS_DIGEST` for the default seed.
- `:197-224` iterates `PRESET_HASHES` and clicks each preset button
  in turn, then waits for `__ATLAS_PREVIEW_LIVE_HASH__` to equal the
  preset's pinned `expectedHash`.

This cannot run in-sandbox (Playwright browser binaries absent),
but the spec file structure is well-formed and the flag/DOM contract
between `src/main.ts` and the spec is consistent. The preview UI's
fallback path on missing-atlas (`src/main.ts:430-435`) is documented
and surfaces an error in `#atlas-preview-error` — a runtime check
the Playwright test does not currently cover (it would be redundant
in the green path, but a `test.skip` or separate spec for the
explicit-failure path could harden it for Phase 4.B).

The `regenerateLive` function (`src/main.ts:402-422`) calls
`validateSeedString(seed)` before `generateAtlas(...)`, so a
malformed user-input seed surfaces a friendly error in
`#atlas-preview-error` rather than crashing the page. Sound.
